Competition Guide

Run large contests by keeping each arena room 1v1 and organizing many rooms through waves, qualification, review, and brackets.

Back to Admin Panel Create Challenge Competitions

Official Rule

A competition may contain many simultaneous 1v1 arena rooms. Each arena room has exactly two active players. For large competitions, players first compete in qualification waves. The highest-ranked players advance to a knockout bracket. Admins supervise groups of rooms, review disputes, and confirm winners.

Create Challenge Choices

Challenge only saves the task without opening a room. Use this when preparing future waves or brackets.

Single 1v1 room creates one normal arena match for two players.

Bulk 1v1 rooms pairs selected players and creates many rooms at once. This is the starting point for waves and qualification rounds.

Auto-pair live rooms opens a competition lobby. Admins make the competition live once; each room starts automatically when two players are paired. Odd-numbered players wait in their assigned room until the next partner joins.

Qualification stage creates official qualifying rooms and stores the results for the competition leaderboard.

Bracket tournament creates a knockout tournament from the selected players. Use this only after the roster is final and the number of players is a supported bracket size.

Qualifier + bracket is for events that begin with qualification records and later advance the top players into a tournament bracket.

Match Creation Fields

Challenge type: choose Image Challenge when players must reproduce a target image, or HTML Challenge when the official target is built from HTML, CSS, and optional JavaScript.

Challenge name, difficulty, and time limit: use names that include the round or wave, such as Grade 10 Wave A. Keep time limits consistent inside one stage so scores are comparable.

Description: write the exact task instructions players should see. Include allowed tools, design constraints, and what counts as a complete submission.

Match Room ID Visibility: Private invite hides the room until admins share it. Assigned-only restricts entry through the competition roster. Public lobby exposes the room to player screens.

Share invite by email: use this when players have verified email addresses and should receive the join link immediately after room creation.

Stage: mark rooms as practice, qualification, review, bracket, or final so admins can tell which rooms affect official results.

Wave or group name: use clear labels such as Wave A, Morning Batch, School 3, or Semifinal Room Set.

Advance rule: write the rule before play starts, for example Top 32 by score, time, and fair play. Do not change this after results are visible unless the director announces the change.

Room mode: 1v1 rooms are the official competition format. Solo score attempts are useful for practice, trials, or qualification when direct pairing is not required.

Allowed attempts: One attempt is best for finals. Best of 2 or Best of 3 is useful for qualification, but every player in the same stage must get the same attempt rule.

Fair-play strictness: Normal is for practice, Strict is for official rounds, and Exam mode is for high-stakes events where inspect tools, source viewing, and suspicious behavior need immediate action.

Tie-break rule: choose higher score then faster submit for most events. Use fewer warnings then score when fair play is more important than speed. Use admin review for finals or disputed rounds.

Auto-start and auto-submit: auto-start is useful when rooms are prepared and players are ready at the same time. Auto-submit should stay on for official timed rounds so unfinished work is still recorded.

Admin assignments: add one admin per line with an admin id, role, and optional room range. Example: 1:director:all or 2:fair_play:1-50.

Large Contest Model

For 1000 students, do not run one giant bracket first. Register everyone, split them into waves, run bulk 1v1 rooms or score attempts, rank by score and fair play, then advance the top 32, 64, or 128 into the official bracket.

Example: 1000 Players

Create a qualification challenge. Run ten waves of about 100 students. Each wave creates 50 parallel 1v1 rooms. After review, use the leaderboard and admin judgment to pick the top 64. Create a 64-player tournament; round one creates 32 rooms, then 16, 8, 4, 2, and 1 final room.

Admin Roles

Tournament Director: creates the contest, starts rounds, resolves final disputes.

Room Admins: watch assigned room ranges such as Rooms 1-20 or Rooms 21-40.

Fair Play Admins: watch camera/mic status, inspect/right-click alerts, suspicious behavior, and disqualifications.

Scoring Admins: review close scores, ties, broken submissions, and manual overrides.

Support Admins: handle login, missing students, wrong rooms, and network problems.

Execution Checklist

1. Create or save the qualification challenge.

2. Select stage: Qualification wave.

3. Add a wave name such as Wave A or Grade 10 Morning.

4. Select an even number of students and create bulk rooms.

5. Start rooms when the wave is ready, or make an auto-pair competition live and let pairs start as they fill.

6. Let auto-scoring rank first, then review disputes and fair-play alerts.

7. Repeat for all waves.

8. Select top players and create a 32, 64, or 128 player bracket.

9. Run every bracket round in parallel.

10. Confirm winners, certificates, and final records.

Room Lifecycle Controls

Make Public or Make Private: switch visibility only when the director is ready for players or spectators to enter. Keep finals private until both finalists are confirmed.

Start: begins the match timer and makes the room live. Use it only after both player names are correct.

Pause: use for verified technical issues, emergency announcements, or director review. Record the reason in the event notes or chat.

Resume: continue only after both players know the match is restarting.

End: closes the room and locks the official result. End rooms after scores are saved and disputes are resolved.

Add 30s: use only for documented issues such as server lag, accidental pause, or director-approved accessibility support. Do not use it to help a losing player.

Share Invite by Email: sends the room link to assigned players when email is available.

Watch Players and Full Monitor: open live supervision views without joining as a player or changing the match result.

Kick player: remove a wrong entrant, disconnected duplicate, or spectator who joined incorrectly. For cheating or rule-breaking, use disqualification instead of a simple kick.

Delete Room: use only for mistaken setup, test rooms, or abandoned rooms that should not count.

Live Monitoring Guide

Use Live Arena Monitor to watch both players' rendered output, code panes, and camera feed where enabled. Admins should not edit player work. They observe, warn, pause, disqualify, or record notes.

Connect live feeds before the wave starts. If a feed is blank, open the full room monitor, refresh the admin panel, and confirm that the player has joined the correct room.

Assign room admins by ranges. A director should watch overall progress while room admins watch detailed behavior in their assigned rooms.

For finals, one admin should watch the full monitor, one should watch chat and fair-play alerts, and one should keep event notes.

Moderation Workflow

Chat Moderation: use this section to search room messages, flagged messages, and admin mentions from one place.

Flag: mark harmful, abusive, suspicious, or rule-breaking chat. Flagging preserves the message for review and notifies the room.

Reply: answer player questions without leaving the admin panel. Keep replies short, neutral, and rule-based.

Open Room: jump from a chat issue into the live match when a message needs context.

Auto-flagged messages: messages with sensitive language may already be marked for review. An admin should still decide whether the issue is accidental, disruptive, or disqualifying.

Admin mentions: treat mentions as support requests. Reply publicly if both players need the answer; open the room if the issue affects scoring or fairness.

Warnings, Kicks, And Disqualification

Warning: use for first-time or minor behavior problems such as right-click attempts, inspect shortcuts, off-topic chat, or confusion about rules.

Kick: use for wrong-room joins, duplicate sessions, disruptive spectators, or a player who must rejoin cleanly before the match starts.

Disqualification: use for cheating, repeated inspect/source attempts, abusive chat, identity problems, refusing admin instructions, or tampering with another player's match.

Before disqualifying, record a short reason. Good reasons are specific: "Repeated inspect shortcut after warning" or "Abusive chat toward opponent." Avoid vague reasons such as "bad behavior."

If a disqualification happens during a tournament match, notify the director so the bracket or qualification leaderboard is reviewed before advancing players.

Competition Rules For Players

Players must join only their assigned room and use only their own account. Sharing accounts, joining for another player, or using another person's device as the competitor is not allowed.

Players must build the requested UI inside the arena editor. External code, copied full solutions, developer tools, inspect shortcuts, view source, and automation that bypasses the challenge are not allowed unless the director explicitly permits them.

Players may ask admins for technical help, but admins cannot provide design answers, code solutions, or strategic hints during official rounds.

Players must keep chat respectful and competition-related. Harassment, threats, offensive language, spam, or attempts to distract an opponent can lead to warning or disqualification.

Players are responsible for submitting before time ends. When auto-submit is enabled, the system submits the current work at the timer end.

Network or device problems must be reported immediately. Admins may pause, add time, restart a room, or schedule a rematch only when the issue is verified and affects fairness.

Scoring And Tie Rules

Automatic score should be treated as the first result, not the final judgment in disputed matches. Admins review close scores, broken submissions, flagged behavior, and technical problems.

When the tie-break rule is higher score then faster submit, compare score first and submission time second. When the rule is fewer warnings then score, fair-play warnings can decide the match before raw score.

For admin review ties, the director should compare final output, task requirements, responsiveness, accessibility, and rule compliance. The decision should be recorded before advancing the winner.

Disqualified players should not advance from a qualification leaderboard or tournament match unless the director reverses the decision after review.

Broadcasts And Announcements

System Broadcast: send room announcements such as "Round starts in two minutes" or "Pause all work." Choose one target room or all active rooms.

Email Everyone: send event updates to registered users. Use verified-only when the event requires reliable delivery records.

Voice Broadcast: use microphone broadcast for urgent live instructions. Keep it short, announce the room or wave first, and avoid giving competitive hints.

For official rounds, broadcast the same instruction to every affected room so no player receives extra information.

Player And Roster Management

Use Add Player before an event when a student is missing from the roster. Keep role set to Player unless the person must access admin tools.

If a password is generated, give it only to the correct student and recommend changing it after login if the app supports that workflow.

Use search and filters to find players quickly. Review username, role, match count, best accuracy, wins, win rate, leaderboard status, and joined date before assigning rooms.

Do not promote a player to admin during a live competition unless the director approves it.

Spectators

Spectators can watch after the first two players occupy the match slots. Admins should keep finals and official rooms private until spectators are allowed.

Remove spectators who distract players, reveal answers, spam chat, or join a restricted room. For public events, assign a support admin to spectator management.

Certificates, Awards, And Records

Use the Certificate Designer before the final event ends. Confirm organization name, competition name, officials, sponsors, signatures, award date, and certificate wording.

After winners are confirmed, award certificates from the player list or tournament page. Do not award certificates before disputes and disqualifications are resolved.

Use award cards for special recognition such as best design, fair play, fastest accurate submission, or outstanding improvement.

Export data after the event if records need to be archived outside the app. Keep challenge names, wave names, winners, disqualifications, and certificate settings consistent for later verification.

Dispute Review Checklist

1. Open the room and review both player outputs.

2. Check chat moderation records and any flagged messages.

3. Check fair-play alerts, right-click or inspect attempts, and admin warnings.

4. Confirm whether the timer, auto-submit, or connection failed for either player.

5. Compare the score with the published tie-break rule.

6. Decide: keep result, adjust by admin review, restart, rematch, or disqualify.

7. Record the reason before advancing the winner or publishing awards.

Emergency Procedures

Wrong challenge: pause or end affected rooms, create corrected rooms, and announce that the first attempt is void.

Wrong player joined: kick the wrong entrant before start, confirm the assigned player, then start or restart the room.

Many players cannot join: keep rooms private, use email invite or room codes again, and delay the wave start instead of starting partial rooms.

Server or network issue: pause affected rooms, broadcast a short status update, and resume only when the director confirms the issue is stable.

Rule dispute: stop advancement for that match, gather logs, and let the tournament director make the final decision.

Current Tools In This App

The Create Challenge tab can save a challenge only, create one room, create many paired 1v1 rooms, open an auto-pair live lobby, create qualification records, and start bracket workflows. The Competitions tab can create database tournaments up to 128 players and advance qualification winners into brackets. The Live Arena lets admins watch rooms, code, output, camera feeds, chat issues, and fair-play incidents.

Operational Safeguards

Admin assignments are saved to the database and shown in My Assigned Rooms. If an admin has active assigned rooms but is not watching any of them, the admin panel shows a warning so supervision duties stay visible.

Every official event should have a director, at least one room admin, and at least one moderation or fair-play admin. Large events should also have support admins for login, room codes, email invites, and spectator questions.

Do not remove existing rooms, scores, or players during a live dispute unless the director has already decided that the record should be void.
